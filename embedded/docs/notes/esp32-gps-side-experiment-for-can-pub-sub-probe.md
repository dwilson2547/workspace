A small ESP32+GPS side experiment is worthwhile to exercise live publish timing, probe propagation, Iggy behavior, and downstream consumers before CAN hardware is wired up. Keep it explicitly separate from the vehicle CAN profile path: publish GPS-derived telemetry as a parallel live source rather than pretending it is CAN input. Repository implementation now lives under can_pub_sub_probe/gps test feed/: esp32_gps_udp_feed/esp32_gps_udp_feed.ino publishes UDP JSON over WiFi once per second when a valid fix exists and every 5 seconds otherwise, carrying latitude, longitude, ground speed, heading, fix quality, satellite count, HDOP, RSSI, device_id, and source_session; wiring defaults are GT-U7 TX to ESP32 GPIO16, optional RX to GPIO17, common ground, and 9600 baud on UART2, with 3.3V power recommended unless the specific GPS breakout explicitly requires 5V. Arduino CLI is installed locally, esp32:esp32 and TinyGPSPlus are installed, and the sketch compiles successfully for fqbn esp32:esp32:esp32. For local smoke tests on WSL2, a UDP listener running inside WSL will not receive packets sent to the Windows host IP because WSL2 NAT does not automatically forward host-side UDP into the guest; the working workaround is to run udp_receiver_example.py on Windows itself (for example via py.exe -3 -u C:\Temp\udp_receiver_example.py --bind 0.0.0.0 --port 5514), which successfully receives packets from the ESP32 at the Windows LAN address.
