For can_pub_sub_probe, start replay-first because in-vehicle CAN connectivity is not available yet; keep live capture as a later milestone. Use the term vehicle CAN profile for the per-vehicle bundle of bus settings and signal definitions so car-specific config stays separate from pipeline logic. Hardware changed from the originally assumed transceiver to TJA1051. The implementation now includes a Python scaffold, 17-byte frame codec, replay source, Impala static vehicle CAN profile, Hop 1 and Hop 2 foundations, Hop 3 routing with versioned routing tables and multi-topic fan-out, a pub-sub-backed router hop runner, a concrete Iggy backend over native TCP, Hop 4 windowed aggregation with derived signals, Hop 5 sink wiring with replaceable raw and aggregate backends, and operator CLI commands for replay-run, inspect-probes, and run-fixture. Fixture coverage now lives under tests/fixtures with committed binary logs, session metadata, and expected summaries for a known-good session plus unknown CAN ID, out-of-range, and malformed DLC scenarios; run-fixture automatically checks expected.json when present, and profile regression tests cover all documented Impala signals. The CLI replay path now applies default validation ranges for documented signals and emits ingest, validate, route, and aggregate diagnostics for real traffic in local regression runs so fixture failures surface as counted reason codes instead of uncaught exceptions. Probe context propagates in pub-sub headers, drop events publish to diagnostics.drop_events, and probe sink suppression still records terminal forwarded_to_sink outcomes so probe tracking can close without polluting downstream storage. The local Iggy stack lives under can_pub_sub_probe/iggy with docker-compose.yml pinned to apache/iggy:0.8.0, host port 8090 only, no HTTP adapter usage, and the router smoke test passes via iggy+tcp://iggy:iggy@127.0.0.1:8090. The older multi-language POC at pub_sub/iggy-poc is useful as historical reference...
